# 模块08：AI 成本经济学

## 为什么必须懂这个

AI 产品的失败，除了"技术不可行"和"用户不需要"，还有第三种常见模式：**成本失控**——产品在技术上成功，用户也喜欢用，但每服务一个用户就亏一笔钱，规模越大亏得越多。这种模式在 LLM 产品中尤为常见，因为 LLM 的推理成本随使用量线性增长，而很多 PM 在立项时没有做过基本的单位经济学测算。

掌握 AI 成本经济学，不是要让 PM 变成财务分析师，而是要让 PM 能在功能设计阶段就排除"经济上不可持续"的方案，以及在 Build vs Buy vs Wrap 的技术选型决策上给出有依据的判断。

## 核心内容

### 推理成本估算：PM 必须能做的基本计算

**LLM 推理成本的构成**：
- **输入 token 成本**：发送给模型的文本量（系统提示词 + 用户消息 + 检索内容）
- **输出 token 成本**：模型生成的文本量
- 通常：输出 token 单价 = 输入 token 单价的 3-5 倍（生成比理解更贵）

**基本估算公式**：
```
单次调用成本 = (输入 token 数 × 输入单价) + (输出 token 数 × 输出单价)
月成本 = 单次调用成本 × 日调用量 × 30
```

**典型场景估算示例**（以 GPT-4o 为参考，价格随市场变化）：
- 单次作文批改：输入 ~2000 tokens（作文内容 + 系统提示词），输出 ~800 tokens
- 单次推理成本约 $0.01-0.03（约 0.07-0.21 元人民币）
- 如果每天有 10 万次批改，月成本约 21,000-63,000 元人民币
- 问：这个成本在你的定价体系中能被覆盖吗？

**成本优化杠杆（PM 可以决策的部分）**：
1. **缩短系统提示词**：精简提示词可显著减少输入 token
2. **限制输出长度**：设置合理的 max_tokens 上限
3. **缓存机制**：对于相同或高度相似的请求，缓存输出避免重复推理
4. **模型降级策略**：高频、简单任务用轻量级模型；复杂任务才用大模型（分层模型策略）
5. **批处理（Batch Processing）**：非实时任务使用批处理 API，通常比实时 API 便宜 50%

### 单位经济学（Unit Economics）：AI 产品的财务健康检验

单位经济学分析的核心问题：**服务一个用户的成本 vs 这个用户带来的价值**。

**AI 产品单位经济学的独特挑战**：
传统 SaaS 的边际成本趋近零（多一个用户几乎不增加成本），但 AI 产品的边际成本随使用量增长。这意味着传统的"增长优先，以后优化成本"策略在 AI 产品上可能导致增长越快亏损越严重。

**健康单位经济学的检验公式**：
```
健康条件：LTV（用户生命周期价值）> 3 × CAC（用户获取成本）+ AI 服务成本总和

AI 服务成本 = 月均推理成本 × 用户生命周期月数
```

**分层成本结构**：并非所有用户都是等成本的。在智学网场景：
- 重度用户（每天大量使用 AI 辅导）：高 AI 成本，但也是最高 LTV 的用户群
- 轻度用户（偶尔查询）：低 AI 成本，但可能也是低 LTV 用户
- 零 AI 功能用户：不产生 AI 成本，但可能流失到使用 AI 的竞品

PM 决策含义：可以用"AI 使用量"作为用户价值分层的维度，对高价值重度用户提供更丰富的 AI 功能，同时控制轻度用户的 AI 成本。

### Build vs Buy vs Wrap：技术路线决策框架

这是 AI PM 最常遇到的三选一决策。每次涉及 AI 能力建设时，PM 都需要在三种路线中做出选择：

**自建（Build）**：从头训练或微调专属模型
- 适合条件：有大量专有数据；对模型行为有极高控制需求；长期来看规模足够大以摊薄训练成本
- 成本：高初始投入（数据、算力、团队）+ 持续维护成本
- 智学网适用：核心优势领域（如特定学科的解题能力）可以考虑

**购买（Buy）**：引入第三方 AI 解决方案或平台
- 适合条件：通用能力，自己没有独特数据或模型优势；快速上市优先
- 风险：供应商锁定（Vendor Lock-in）；供应商停服的风险；数据隐私（学生数据发给第三方）
- 智学网注意点：教育数据涉及未成年人，购买第三方 AI 服务前必须评估数据处理合规性

**封装调用（Wrap）**：通过 API 调用基础大模型（GPT-4、Claude、文心一言等），在其上构建业务逻辑
- 适合条件：能力需求在基础大模型覆盖范围内；专有数据不足以支撑自建；需要快速验证
- 优点：快速、低初始成本、能跟随基础模型能力升级
- 风险：成本随用量线性增长；基础模型的能力上限就是你的上限；价格受供应商控制

**决策树**：
```
是否有专有数据 + 明确的性能差距 + 足够规模？
  → 是 → Build（评估自建 ROI）
  → 否 ↓
是否有满足需求的成熟第三方解决方案？
  → 是 → Buy（评估数据合规和供应商风险）
  → 否 → Wrap（用 API 构建，保持灵活性）
```

## 实战场景

**场景：为智学网 AI 解题辅导估算经济可行性**

假设：
- 目标用户：50万付费学生，30%月活跃（15万人/月）
- 平均每活跃学生每月使用 AI 辅导：20次/月
- 每次平均调用：输入2000 tokens + 输出1000 tokens
- 模型：GPT-4o 价格（输入 $2.5/M tokens，输出 $10/M tokens）

**月成本计算**：
- 输入成本：15万人 × 20次 × 2000 tokens / 100万 × $2.5 = $15,000/月
- 输出成本：15万人 × 20次 × 1000 tokens / 100万 × $10 = $30,000/月
- 总计：$45,000/月 ≈ 32万元人民币/月

**可行性判断**：
- 如果 AI 辅导是高级包功能（+50元/月），15万活跃用户贡献 750万元/月
- AI 成本占收入比：32/750 ≈ 4.3%，健康（通常可接受上限为10-15%）
- 如果 AI 辅导是免费功能：每月净成本 32 万元，需要在其他地方找回这个成本

**结论**：在高级付费功能场景下，经济可行。应避免将高成本 AI 辅导功能放入免费层。

## 决策框架

**AI 功能成本可行性的3步评估**

1. **估算单次成本**：基于 token 预算计算单次 API 调用成本
2. **估算月规模成本**：基于预期用户量和使用频率计算月成本总额
3. **对比收入贡献**：这个功能直接或间接产生的收入是否足以覆盖成本（建议成本/收入比 < 15%）

## 常见误区

- **误区1：AI 成本会随时间自动降低，现在不需要担心** → 正确理解：虽然 LLM 的单位价格在下降，但如果同期使用量快速增长，总成本仍可能失控。成本管理需要从第一天开始，而非等到规模大了再优化。

- **误区2：Wrap 模式没有竞争壁垒，不如自建** → 正确理解：Wrap 模式结合专有数据、深度工作流整合和精心设计的 Prompt 工程，可以建立足够强的差异化。并非所有产品都需要自建模型。

- **误区3：成本优化是工程团队的事，和 PM 无关** → 正确理解：许多成本优化机会来自产品设计决策——减少不必要的上下文长度、选择合理的模型分级策略、设计合理的缓存逻辑。这些都是 PM 与工程团队共同决策的范畴。[Source: Lipenkova, 2024]
