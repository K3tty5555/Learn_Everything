# 模块09：MLOps 与运维现实

## 为什么必须懂这个

AI 产品的生命周期不在上线时结束，而是在上线时开始。传统软件上线后如果没有 Bug 就能稳定运行；ML 系统不同——即使代码没有变化，模型性能也可能随时间退化（因为现实世界的数据分布在变化）。不懂运维现实的 PM 会在两个场景下栽跟头：一是上线后盲目自信、疏于监控，导致性能退化问题长期未被发现；二是无法与工程团队就"什么时候需要重新训练模型"建立清晰的决策框架。

## 核心内容

### ML 技术债务：代码之外的隐性负担

Sculley 等人在 2015 年的 NeurIPS 论文中提出"ML 系统中的隐性技术债务"概念，揭示了 ML 系统比传统软件积累更多难以偿还的技术债务的原因。[Source: Sculley et al., Hidden Technical Debt in ML Systems, NeurIPS 2015]

**PM 需要警惕的五类 ML 技术债务**：

**1. 纠缠（Entanglement）债务**
ML 系统中，改变任何一个输入特征都可能影响所有其他特征的权重。这意味着你以为"只是调整了一个功能的数据收集方式"，可能无意中影响了整个模型的行为。

**2. 隐藏反馈循环（Hidden Feedback Loops）**
模型的预测影响用户行为，用户行为产生训练数据，训练数据影响下一版模型。这种循环可能导致模型自我强化偏见（如：推荐系统只推送热门内容，导致小众但高质量内容永远得不到曝光机会）。

**3. 数据依赖（Data Dependency）**
ML 系统对数据管道的依赖比传统软件对代码库的依赖更脆弱。上游数据源的格式变化、数据质量下降、数据采集逻辑变更，都可能在不触发任何代码警报的情况下悄悄损害模型性能。

**4. 配置复杂性（Configuration Complexity）**
ML 系统有大量超参数和配置选项。随着时间推移，没有人能完全理解每个配置的意图。"不知道为什么但改了这个参数效果就好了"是 ML 团队中的常见情形。

**5. 模型腐化（Model Rot）**
即使所有代码和配置都没变，现实世界数据的漂移会导致模型性能逐渐退化。

PM 的核心职责：在产品路线图中明确 ML 技术债务偿还的优先级，不让工程团队无休止地堆砌新功能而从不偿还债务。

### 模型漂移监控：PM 需要定义的触发条件

**数据漂移（Data Drift）** vs **概念漂移（Concept Drift）**：

- **数据漂移**：输入数据的分布发生变化（例如：学生群体结构变化，新增大量低年级用户，而模型是在高年级数据上训练的）
- **概念漂移**：输入和输出之间的关系发生变化（例如：教学大纲改革，知识点关联关系变了）

**PM 需要定义的监控指标和重训练触发条件**：

| 场景 | 监控指标 | 触发条件（示例） |
|------|---------|--------------|
| 推题系统 | 推荐题目完成率 | 连续2周低于70%（基线值下降>10%） |
| 作文批改 | 教师修改 AI 评分的比率 | 超过25%（说明 AI 评分质量下降） |
| 对话辅导 | 用户对 AI 回复的"无用"标记率 | 超过15% |
| 学情诊断 | 诊断结论与后续实际成绩的相关性 | 相关系数低于0.6 |

PM 职责：这些触发条件不是工程师自己设定的，而是 PM 根据产品质量标准和业务影响来定义的，再由工程团队实现监控和告警。

[Source: Huyen, Chip. Designing Machine Learning Systems. O'Reilly, 2022, Chapter 8]

### MLOps 基本概念：PM 视角

**机器学习运维（MLOps）** 是将 ML 模型开发和生产部署进行有效管理的工程实践。PM 不需要实现 MLOps 工具链，但需要理解其对产品发布节奏的影响。

**PM 需要知道的 MLOps 关键流程**：
- **特征工程管道**：数据从原始形态到模型输入格式的转换流程。管道越复杂，出错的地方越多。
- **模型注册与版本管理**：类比代码版本控制，模型也需要版本管理（能快速回滚到上一个稳定版本）。
- **影子部署（Shadow Deployment）**：新模型在生产环境并行运行但不影响用户，用于在真实流量下验证新模型表现。
- **金丝雀发布（Canary Deployment）**：新模型先对少量用户（1-5%）生效，验证无问题后逐步扩大比例。

这些实践对 PM 的意义：模型更新不是"改一下代码重新部署"那么简单，需要数据收集周期 + 训练时间 + 评估时间 + 安全部署时间，整个周期可能需要数周。PM 在规划路线图时必须考虑这个周期。

## 实战场景

**场景：智学网 AI 推题系统出现性能退化**

症状：本学期开学后，学生对 AI 推题的完成率下滑了12%，"题目太难"的反馈增加。

**PM 的诊断框架**：
1. 是数据漂移吗？（新学期学生年级结构是否变化？新增低年级用户？）
2. 是概念漂移吗？（新教材/大纲修订后，知识点图谱是否需要更新？）
3. 是特征管道问题吗？（上游数据收集逻辑是否有变化？）
4. 是反馈循环问题吗？（上学期的推题结果是否产生了有偏差的训练数据？）

根据诊断结果，触发对应的改进行动（数据增补、重训练、知识图谱更新）。

## 决策框架

**模型重训练决策树**

```
监控指标触发告警？
  → 否 → 维持现状，继续监控
  → 是 ↓
性能退化是否影响关键用户体验指标？
  → 否 → 记录观察，继续跟踪
  → 是 ↓
能否通过 Prompt 调整或规则修复？
  → 是 → 快速热修复（不需要重训练）
  → 否 → 触发模型重训练流程
```

## 常见误区

- **误区1：上线后不出报错就代表模型正常** → 正确理解：ML 系统的"silent failure"（静默失败）极为常见——代码正常运行，但输出质量在悄悄下降。没有主动监控，永远发现不了漂移问题。[Source: Sculley et al., NeurIPS 2015]

- **误区2：发现性能退化就立刻重新训练** → 正确理解：重训练有成本和风险（新模型可能引入新问题）。应先诊断根因，确认是漂移问题而非数据管道 Bug；同时评估退化的严重程度是否值得承担重训练风险。

- **误区3：MLOps 是工程问题，和 PM 无关** → 正确理解：模型发布周期、监控指标定义、重训练触发条件——这些都需要 PM 参与定义。工程师知道"怎么做"，PM 决定"为什么做"和"什么时候做"。
