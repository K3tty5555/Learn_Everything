# 设计文档：模块优先架构（Module-First Architecture）

**日期：** 2026-03-31
**状态：** 已确认，待实施
**背景：** 用户在学习 AI PM 领域时发现 Agent 等关键概念在知识库中几乎缺失，追溯根因后决定对 Learn Everything 系统进行架构级重设计。

---

## 问题诊断

### 直接问题
`domains/ai-product-manager` 经历了完整的 P1-P4 流程后，Agent 这一 2026 年 AI PM 的核心概念在知识库中只有一行词典条目，无法支撑学习需求。

### 根本原因

**1. 知识库以论文为中心，不以学习者需求为中心**
P2 的产出结构反映的是「这10篇论文说了什么」，而不是「学习者需要掌握什么」。Agent 在那10篇 PM 学术书籍里只是配角，所以 KB 里也是配角。

**2. P1 假设所有领域的权威知识都在学术论文里**
这个假设对理论型领域成立，但对 AI PM 这类混合型领域是错的。Agent 的最佳实践在 Anthropic 博客、LangChain 文档、工程师实战总结里，不在 CHI 论文里。P1 找不到这些来源，P2 自然写不出来。

**3. 产出是档案库，不是学习材料**
25+ 个分散文件（papers/ + concepts/ + qa/ + cheatsheet + ...）对于学习者来说认知负担极高，没有叙事，没有递进，不知道从哪开始读。

**4. zero-hallucination 原则的副作用**
每个 claim 必须有 source，但论文没覆盖到的知识就不写。保证了质量，但限制了深度。

---

## 设计目标

> **Learn Everything 产出的应该是一本为学习者定制的书，而不是一个档案库。**

- 用户读完 `published/` 后，对该领域有系统性、可用的知识
- 每个重要概念都有足够深度，不再出现「只有一行」的情况
- 知识来源匹配领域的实际知识形态，不强求一切来自学术论文

---

## 核心架构变化

### 原则转变

```
旧：找论文 → 写笔记 → 提炼概念 → 发布
新：定义学习目标 → 找最佳来源 → 构建模块 → 发布
```

### 新目录结构

```
domains/{slug}/
  curriculum.md        ← P1 新产出：课程大纲 + 模块定义
  sources.md           ← P1 新产出：分类来源清单（替代 skeleton + researchers）
  modules/             ← P2 核心产出（替代 papers/ + concepts/）
    01-{slug}.md
    02-{slug}.md
    ...
  learning-path.md     ← P4 产出，直接链接模块
  meta.md
  published/           ← P5 直接编译 modules/ → DOCX
```

---

## P1 重写：领域分析 + 课程设计

### 现状
找 3-5 个研究者 + 10 篇高引论文，输出 skeleton.md + researchers.md。

### 新设计：三步流程

**Step 1 — 领域知识形态判断**

分析该领域的权威知识分布：

| 形态 | 说明 | 示例领域 |
|------|------|---------|
| 学术型 | 核心知识在期刊/会议论文 | 机器学习理论、认知科学 |
| 实践型 | 核心知识在从业者书籍/博客/案例 | 产品管理、创业策略 |
| 技术文档型 | 核心知识在官方文档/工程博客 | Prompt Engineering、框架使用 |
| 混合型 | 多种形态并存，按比例分配 | AI Product Manager |

输出：领域类型 + 来源策略（各类型来源的比例与重点）

**Step 2 — 课程大纲设计**

定义 8-12 个学习模块：
- 模块名称与核心问题（「学完这个模块，我能做什么」）
- 模块间的依赖关系（哪些需要先学）
- 每个模块的深度要求（概览 / 掌握 / 精通）

输出：`curriculum.md`

**Step 3 — 来源发现（按类型分类）**

按领域知识形态，分类找来源：

```
学术来源：    顶尖研究者 + 关键论文（保留原有逻辑）
实践来源：    从业者 + 畅销书籍 + 权威博客 + 行业案例
技术来源：    官方文档 + 工程博客 + GitHub 优质项目
```

每个来源标注：覆盖哪些模块（source → module 映射）

输出：`sources.md`（替代原来的 skeleton.md + researchers.md）

## P2 重写：按模块并行构建内容

### 现状
并行分析每篇论文，每篇论文输出一个笔记文件，论文是结构主体。

### 新设计

输入：`curriculum.md` + `sources.md`

**并行构建每个模块：**
1. 读取该模块的来源映射
2. 综合多来源（学术 + 实践 + 技术文档）
3. 撰写 1500-3000 字的完整模块文档
4. 每个 claim 标注具体来源（zero-hallucination 原则保留）

**每个模块文件的固定结构：**

```markdown
# 模块 N：{模块名}

## 为什么 PM 必须懂这个
（1-2段，建立学习动机）

## 核心内容
（深度叙述，1000-2000字，有小节划分）

## 实战场景
（结合用户具体场景的案例，如智学网）

## PM 的决策框架
（这个概念如何影响你的日常工作和决策）

## 常见误区
（学习者容易犯的认知错误）

## 来源
[Source: ...]

## 延伸阅读
（如需深入，看这些）
```

输出：`modules/01-xxx.md` ... `modules/N-xxx.md`

## P3-P5 适配调整

### P3 learn-challenge
- 改为**逐模块挑战**，而非对整个 domain
- 每个模块单独经历多专家挑战，识别该模块的盲点
- 挑战发现的缺失内容直接更新到对应模块文件

### P4 learn-path
- 路线图基于 `curriculum.md` 中的模块依赖关系生成
- 每个学习步骤**直接链接到对应模块文件**
- 不再出现「第3周学 Agent」但点进去没内容的情况

### P5 learn-publish
- 输入直接是 `modules/` 目录
- 按模块顺序编译成 DOCX（模块 = 书的章节）
- 几乎零转换成本，published/ 就是 modules/ 的格式化版本

---

## 对现有域的影响

现有 `domains/ai-product-manager` 需要用新架构重跑 P1 + P2。
- 原有的 papers/、concepts/ 作为参考资料保留，不删除
- 新增 curriculum.md、sources.md、modules/ 目录
- meta.md 更新以反映新结构

---

## 成功标准

1. 用新架构跑完 `ai-product-manager` 后，Agent 有完整的独立模块（≥1500字，含实战场景和 PM 决策框架）
2. 用户读完 `published/` 后，不需要回到 CC 补基础知识，只用 CC 做深化和答疑
3. 模块文件可以直接阅读，不需要在多个文件间跳转拼图
